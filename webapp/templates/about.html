Thesis Overview

This project is an MSc thesis that investigates how cloud design patterns can be applied to build a scalable, provider-agnostic Analytics as a Service (AaaS) system. The chosen domain is medical image classification — specifically the automated detection and categorisation of brain tumors from MRI scans into four classes: Glioma, Meningioma, No Tumor, and Pituitary.

The thesis follows a five-phase methodology covering system design, environment setup and data preparation, infrastructure and backend development, deployment and performance measurement, and finally critical analysis and documentation. Each phase is fully reproducible: all infrastructure is declared as code using Terraform, and all compute workloads are packaged as Docker container images, ensuring the system can be ported to any compatible cloud provider with minimal rework.

The project critically evaluates existing cloud-based machine-learning architectures, identifies their limitations — such as over-reliance on vendor-specific services, lack of portability, and poor lifecycle management — and demonstrates how a cloud-design-pattern approach directly addresses those gaps.

Cloud Design Patterns Applied

The central thesis contribution is the identification and application of proven cloud design patterns to an analytics workload. Rather than building a monolithic application tightly coupled to a single provider, the system separates concerns into independently deployable, replaceable modules — a pattern known as the Strangler Fig or Backends for Frontends approach. Each pattern is applied deliberately:

Serverless / Event-Driven Pattern

AWS Lambda handles inference requests triggered by API Gateway HTTP events. No servers to manage; the function scales from zero to thousands of concurrent executions automatically. The pattern is ideal for bursty, short-lived workloads such as on-demand image classification.

Container-Based Microservice Pattern

Both the Lambda function and the ECS Fargate service run Docker container images stored in Amazon ECR. Containers encapsulate the entire runtime environment — Python, PyTorch, model weights, and the inference handler — making the workload fully portable and reproducible regardless of the underlying host.

VM-Based Deployment Pattern

An Amazon EC2 instance (AL2023, t3.medium) running Docker provides a third deployment target, demonstrating that the same Docker image can be deployed across serverless, container-orchestrated, and VM environments — the very essence of cloud-agnostic design.

Infrastructure as Code Pattern

All AWS resources — API Gateway, Lambda, ECS, EC2, S3, DynamoDB, ECR, IAM, KMS, and Secrets Manager — are provisioned exclusively through 9 modular Terraform configurations. One command (terraform apply) reproduces the entire production environment from scratch in any AWS region or account.

Model Registry Pattern

Trained model artefacts are stored in Amazon S3 (versioned, KMS-encrypted) and registered in a DynamoDB metadata table with a composite key (model_name + version). Any compute backend resolves the correct artefact at runtime via a GET /models/{name} lookup — decoupling model lifecycle management from inference execution.

Security by Default Pattern

Every resource applies least-privilege IAM roles, KMS encryption at rest (S3, DynamoDB, CloudWatch Logs), IMDSv2 enforcement on EC2, and secret rotation via AWS Secrets Manager. Security is not bolted on — it is embedded directly in the Terraform modules.

Docker — The Portability Layer

Docker is the portability backbone of this project. Every compute workload is packaged as a Docker container image, pushed to Amazon ECR, and pulled at runtime by whichever deployment target is active. This means the exact same model code, Python environment, and dependencies run identically whether the container is executed by:

  • AWS Lambda — using Lambda's container image runtime (up to 10 GB image, 3008 MB RAM).
  • Amazon ECS Fargate — a fully managed container orchestration service with auto-scaling.
  • Amazon EC2 — a t3.medium virtual machine running Docker Engine on Amazon Linux 2023, serving the inference HTTP API directly on port 8080.

This three-way deployment approach is the empirical proof of the cloud-agnostic design pattern: the same artefact (Docker image) is consumed by three structurally different compute paradigms without any code change to the inference service itself. The Terraform modules/ecr module manages the image registry, lifecycle policies, and access permissions for both the Lambda and ECS images.

Dataset and Deep Learning Model

The classification model is trained on the publicly available Brain Tumor MRI Dataset (Masoud Nickparvar, Kaggle), which contains pre-split training and testing sets across four classes. A MobileNetV2 backbone (pre-trained on ImageNet) is fine-tuned using a two-phase transfer-learning strategy in PyTorch:

Phase 1 — Feature Extraction

The MobileNetV2 convolutional base is frozen. Only the new classification head is trained for several epochs to learn domain-specific features from MRI image distributions.

Phase 2 — Fine-Tuning

The upper layers of the backbone are unfrozen and trained jointly with the head at a lower learning rate, allowing the model to adapt low-level features to the MRI domain. Final test accuracy: 94.56%.

The trained model is exported in both PyTorch (.pt) and ONNX formats, uploaded to the S3 model bucket via scripts/upload_model.py, and registered in DynamoDB via scripts/register_model.py. At inference time, any compute backend downloads the .pt file from S3 on cold start, loads it with torch.load(), and serves predictions through a lightweight Flask HTTP API.

API Contract

The inference API is exposed through AWS API Gateway (HTTP API v2) and proxied to the active Lambda backend. The contract is kept minimal and cloud-agnostic:

POST /predict

Input: multipart/form-data

metadata: {"model": "mri"}
image: <binary JPG/PNG>

Output:

{"classification": "glioma",
 "confidence": 0.984,
 "model": "mri"}
GET /models  ·  POST /models

List all registered models or register a new one.

GET /models/{name}

Retrieve metadata for a specific model.

DELETE /models/{name}/{version}

Deregister a specific model version.

Reproducibility

Full reproducibility was a primary design constraint. Any researcher with valid AWS credentials can recreate the entire system by:

  1. Cloning the repository from GitHub.
  2. Running model/train_model.py to train the MobileNetV2 classifier.
  3. Running scripts/upload_model.py and scripts/register_model.py to publish the model artefact.
  4. Running scripts/build_and_push.py to build and push Docker images to ECR.
  5. Running terraform apply inside terraform/ to provision all AWS resources.
  6. Launching the Flask web app with python webapp/app.py to interact with the live API.
