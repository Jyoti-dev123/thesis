Overview

What Is This Project?

This thesis project designs and implements a fully cloud-agnostic Analytics as a Service (AaaS) platform that classifies Brain Tumor MRI scans using deep learning. The system exposes a single REST API endpoint — POST /predict — that accepts an MRI image and returns a JSON prediction from a trained MobileNetV2 convolutional neural network hosted on AWS.

The core innovation of this project is its emphasis on infrastructure portability. By encoding every AWS resource — API Gateway, Lambda, ECS, S3, DynamoDB, IAM, KMS, and Secrets Manager — in modular Terraform configurations, the entire cloud environment can be torn down and reproduced in minutes, or migrated to a different provider by swapping provider modules. This addresses a key gap in existing cloud ML deployments: hard-wired, vendor-locked infrastructure that cannot be easily moved or reproduced.

Three distinct compute architectures are provisioned and benchmarked side-by-side: AWS Lambda (serverless container), ECS Fargate (managed container), and Elastic Beanstalk (platform-as-a-service). This multi-architecture comparison gives practitioners evidence-based guidance on which deployment model best fits their latency, throughput, and cost requirements.

The end product is a fully functional, secured, tested, and documented system — complete with a Flask web interface for live demonstrations, performance benchmarking tools, and all code published openly on GitHub for reproducibility.

Research Objectives

  • 1 Design a reusable, cloud-agnostic AaaS pattern using Infrastructure-as-Code (Terraform).
  • 2 Train a CNN-based classifier (MobileNetV2) on 7,000+ Brain Tumor MRI images across 4 classes.
  • 3 Deploy, test, and benchmark three compute architectures: Lambda, ECS Fargate, and Beanstalk.
  • 4 Implement end-to-end security: encryption at rest (KMS), API key auth, and least-privilege IAM.
  • 5 Critically evaluate AaaS pattern limitations against existing approaches in the literature.

Classification Target — 4 Brain Tumor Classes

High Risk
Glioma

Originates from glial cells in the brain or spine. Accounts for ~33% of all brain tumors. Includes glioblastoma. Often aggressive and requires combined treatment.

Moderate
Meningioma

Arises from the meninges surrounding the brain. The most common primary brain tumor (~30%). Most are benign and slow-growing; treatment often surgical.

Benign
No Tumor

Scan shows no visible abnormality. Serving as the negative control class, it is critical for minimising false positives in clinical environments.

Moderate
Pituitary

Develops in the pituitary gland at the base of the brain. Usually benign adenomas that can affect hormone regulation. Represents ~15% of all brain tumors.

Methodology

Five-Phase Research Methodology

The project follows a structured five-phase methodology that incrementally builds, deploys, tests, and evaluates the AaaS system. Each phase produces tangible artefacts — code, infrastructure, data, or documentation — that feed directly into subsequent phases.

Phase 1

Scoping, Design & Architectural Exploration

Reviewed existing cloud ML deployment patterns and identified limitations: vendor lock-in, lack of IaC, poor model lifecycle management. Defined the API contract (POST /predict), selected three deployment architectures (Lambda, ECS, Beanstalk), and designed the Terraform module structure. Produced architectural ADRs, security design decisions, and a comparison matrix of deployment trade-offs.

Phase 2

Environment Setup & Data Preparation

Configured Python virtual environment, installed PyTorch 2.9.1 + CUDA 12.8, Terraform v1.5+, AWS CLI, and Docker. Downloaded the Kaggle Brain Tumor MRI Dataset (7,023 images). Implemented preprocess.py to resize images to 224×224 RGB, normalise pixel values, and apply augmentation (random flips, rotation, colour jitter). Trained MobileNetV2 using two-phase transfer learning — first training only the classifier head, then fine-tuning all layers — achieving 94.56% test accuracy.

Phase 3

Infrastructure Development & Backend

Authored 8 Terraform modules. Provisioned API Gateway, Lambda (3008 MB, 120s timeout, container image), ECS Fargate service, two S3 buckets, DynamoDB metadata table, ECR repositories, IAM roles, KMS key, and Secrets Manager. Implemented Lambda handler and ECS Flask inference service. Built model management scripts to upload, version, and register models. Ran terraform apply to deploy all infrastructure.

Phase 4

Deployment, Testing & Benchmarking

Deployed Lambda and ECS Docker images to ECR and applied infrastructure. Ran functional tests sending 20 real MRI images to /predict, achieving 85% pass rate (17/20 correct). Measured cold-start latency (1,121 ms), warm average (1,147 ms), P90 (1,187 ms), and throughput (0.87 req/s). Estimated cost at $0.056 per 1,000 requests on Lambda. Compared Lambda vs ECS vs Elastic Beanstalk on seven dimensions.

Phase 5

Analysis, Documentation & Reproducibility

Analysed system performance, scalability, and cost-efficiency. Produced architecture diagrams (overview and detailed, in PlantUML). Built a Flask web application with Bootstrap 5 UI for live demonstration and integrated a benchmarking page with Chart.js visualisations. Published all code — 53 files, 4,975 insertions — on GitHub. Documented known limitations and future improvement directions.

Key Insight

Cloud-Agnostic by Design

By encoding all AWS resources in Terraform modules with parameterised inputs, the system's compute layer (Lambda/ECS/Beanstalk) can be swapped without touching application code. The API contract and model storage interface remain stable — only the provider block and module implementations change. This is the core academic contribution of the thesis.

Step 3 — Model Training (MobileNetV2, PyTorch + CUDA)

Phase 2

A pre-trained MobileNetV2 (ImageNet weights) was fine-tuned on the MRI dataset using a two-phase transfer learning strategy. In Phase A, only the final classifier head was trained (10 epochs, learning rate 1e-3) while all backbone layers were frozen. In Phase B, the entire model was unfrozen and trained end-to-end at a reduced learning rate (1e-4, 10 more epochs) to allow fine-grained feature adaptation. Cross-entropy loss, Adam optimiser, and a step learning-rate scheduler were used. Training ran on an NVIDIA GPU with CUDA 12.8. The trained model was exported in two formats: brain_tumor_model.pt (PyTorch native) and brain_tumor_model.onnx for runtime-agnostic deployment.

ML Model

Machine Learning Model

The classification model is a MobileNetV2 convolutional neural network pre-trained on the ImageNet dataset (1.4M images, 1,000 classes). MobileNetV2 was chosen for its excellent trade-off between accuracy and inference speed, making it suitable for deployment in latency-sensitive cloud environments. Its depthwise separable convolutions reduce computation by a factor of roughly 8–9× compared to standard convolutions while maintaining competitive accuracy.

The pre-trained backbone's feature extraction capabilities were leveraged through transfer learning. The ImageNet classifier head was replaced with a new fully-connected layer producing four output logits (one per tumour class). Training proceeded in two phases: first, only the new head was trained to warm up the classifier weights; second, all layers were unfrozen for end-to-end fine-tuning at a 10× lower learning rate to preserve the backbone's general features while adapting them to MRI texture.

The final model achieves 94.56% accuracy on the 1,311-image held-out test set — strong performance given the visual similarity between some tumour types (particularly glioma and meningioma in T1 MRI). The model weights are stored in PyTorch native format (.pt) and additionally exported to ONNX for runtime-agnostic deployment scenarios.

Infrastructure as Code

Terraform Infrastructure

All AWS infrastructure is defined declaratively in Terraform. The modular design means each AWS service family is fully encapsulated — compute can be swapped between Lambda, ECS, and Beanstalk by changing a single module reference without affecting storage, security, or networking.

The root module (terraform/main.tf) acts as a wiring harness. It instantiates each child module and threads outputs from one into inputs for the next — for example, the KMS key ARN from the security module is passed to both the s3 and dynamodb modules to enable at-rest encryption with the same customer-managed key. This pattern enforces DRY (Don't Repeat Yourself) principles across the configuration.

Variable files (terraform.tfvars) externalise all environment-specific values: AWS region, account ID, project name, and feature flags (e.g., whether to enable Elastic Beanstalk). This means the same Terraform codebase can deploy identically to dev, staging, and prod environments by switching a single tfvars file — a core property of a cloud-agnostic pattern.

The security module provisions a KMS Customer Managed Key with rotation enabled. All S3 bucket objects, DynamoDB table data, and CloudWatch log groups are encrypted using this key. The IAM module creates execution roles with the minimum required permissions — each role grants only the S3 bucket and DynamoDB table ARN it actually needs to access, implementing true least-privilege access control.

API

API Contract

The system exposes a single endpoint that serves as the stable contract between any client and any backend compute implementation. Keeping the API surface minimal means clients need no knowledge of whether prediction is performed by Lambda, ECS, or Elastic Beanstalk — the response schema is identical in all cases.

Requests are authenticated via an API key supplied in the x-api-key HTTP header, managed through AWS API Gateway usage plans. The image and metadata are sent as multipart/form-data to allow binary image payloads alongside structured JSON metadata — matching how clinical systems typically submit medical images.

The response always contains a classification field with one of the four valid class names. When served through the web application, additional fields such as confidence and latency_ms are appended by the Flask proxy layer to enrich the UI experience without modifying the core API contract.

Architecture Comparison

Three AWS compute architectures were provisioned and evaluated against five criteria:

Criterion Lambda (Serverless) ECS Fargate (Container) Elastic Beanstalk (Managed)
Cold-start latency 1,121 ms None (always warm) Minimal
Warm latency 1,147 ms avg Lower (persistent container) Similar to ECS
Scaling Instant, automatic Target-tracking auto-scale Managed, slower scale-out
Cost model Pay per invocation ($0.056/1K) Pay per vCPU/memory per second EC2 instance hours
Best use case Sporadic, event-driven inference High-throughput, sustained load Simplified deployment, teams avoiding containers
Operational burden Minimal Moderate Moderate

System Strengths

  • Full Infrastructure-as-Code — entire environment reproducible in minutes with terraform apply.
  • Cloud-agnostic design — compute layer swappable without changing application code or API contract.
  • Defence-in-depth security — KMS encryption at rest, API key auth, least-privilege IAM, Secrets Manager.
  • Model lifecycle management — versioned S3 storage, DynamoDB metadata registry, upload/register scripts.
  • High classification accuracy — 94.56% on held-out test set with MobileNetV2 transfer learning.
  • Full reproducibility — all code, config, and scripts publicly available on GitHub.

Limitations & Future Work

  • Lambda cold-start (~1.1s) is non-trivial for real-time clinical use; provisioned concurrency would mitigate this.
  • The system is AWS-specific at the provider level — true multi-cloud portability would require provider-abstraction modules for Azure/GCP.
  • 85% functional test pass rate (not 100%) reflects edge cases where confusable tumour morphologies are misclassified.
  • No CI/CD pipeline — GitHub Actions would automate image build, push, and deployment on every commit.
  • No real-time model drift monitoring — adding CloudWatch metrics for per-class confidence distributions would detect data shift.
  • Elastic Beanstalk architecture was designed but performance not benchmarked in this iteration.
